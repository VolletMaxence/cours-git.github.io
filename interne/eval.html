[Concept 1] A quoi sert un tag ? 
	a) à tirer une nouvelle ligne de développement sur un projet
	b) à déplacer les modifications de la copie locale vers le dépôt
	c) à retrouver une modification précise dans l'historique
	d) à fusionner deux branches
	e) à prendre une "photo" de l'état du dépôt à un moment donné et à lui affecter une étiquette

[Concept 2] Dans quel cas d'utilisation n'utilisez vous PAS de branche ? 
	a) pour un éditeur, produit légèrement différent d'un client à l'autre
  	b) plusieurs versions d'un même projet (branche de développement et de maintenance par exemple)
  	c) séparation des tests et du code
  	d) tentative de refactoring (restructuration) du code ou nouvelle fonctionnalité fortement impactante pour le code existant
  	e) nouvelle fonctionnalité ou correction de bogue

[Concept 3] Dans cette liste de ressources, laquelle stockez vous en configuration ?
	a) classes java
	b) image statique d'un site Web
	c) code source généré
	d) fichiers HTML générés par l'outil javadoc
	e) spécifications et dossiers de conception 	

[Concept 4] Concernant les conflits, quelle proposition est FAUSSE 
	a) un conflit se produit lorsque plusieurs développeurs modifient de façon concurrente la même zone de texte d'un fichier
	b) une modification concurrente ne provoque pas toujours de conflits, le VCS sait fusionner les modifications non conflictuelles
	c) un conflit nécessite toujours une intervention manuelle
	d) il est possible de valider (commiter) une modification en conflit avec le dépot mais il faudra résoudre ce conflit ultérieurement
	e) pour résoudre un conflit entre ses modifications et celles de Bob, Alice peut conserver certaines de ses propres modifications et intégrer certaines de Bob

[SVN 1] Vous modifiez un fichier source et vous tentez un commmit mais le serveur SVN vous informe que le fichier a déjà été modifié entre temps. Que faites-vous ? 
	a) Vous forcez le commit avec l'option -f pour vous assurer que vos modifications sont bien prises en compte
	b) Vous créez une nouvelle branche pour ne pas entrer en conflit avec la version du dépôt 
	c) Vous comparez votre version à celle du dépôt, vous reportez manuellement les différences et vous tentez à nouveau le commit
	d) Vous faites un 'update' puis corrigez d'éventuels conflits puis pour tentez à nouveau le 'commit'
	e) Vous écrasez votre fichier avec la dernière version du dépôt et vous réappliquez vos modifications
